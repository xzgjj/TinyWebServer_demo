# v7

v7 的核心目标，把 v6 的 Reactor 框架真正拉到工程可用状态：

- 状态机可验证
- 资源边界可控
- 错误路径可追踪
- 配置与观测可落地
- 安全校验进入主链路

一句话：**v7 = 从“能跑”升级到“、可回归、可演进”。**

---

## v7 相比 v6 的变化重点

，v7 是按阶段把蓝图逐步落地：

1. **v7.1**：Connection 完整状态机 + 资源边界控制  
2. **v7.2**：统一错误码体系 + 超时管理  
3. **v7.3**：JSON 配置 + 指标系统 + 结构化日志  
4. **v7.4**：请求安全验证 + HTTP/1.1 完整性增强（Keep-Alive、条件请求）+ HTTP/2 基础模块接入  

---

## v7 的工程视角（核心模型）

Reactor 在 v7 的真实定义：

**事件（epoll） -> 状态（Connection FSM） -> 行为（读/写/验证/响应/关闭）**

你不能再把它看成“单纯的 epoll 封装”，它现在是“可治理连接生命周期系统”。

### 连接主状态（简化）

```text
[Connecting]
    |
    v
[Connected]
    |
    | EPOLLIN
    v
[Reading] -> [Processing] -> [Writing]
    ^                         |
    |------ keep-alive -------|
    |
    +------ error/timeout ----> [Closing] -> [Closed]
```

---

## 当前实现状态（结合仓库现状）

### 已落地

1. **Connection 状态机**
   - 显式状态枚举与状态迁移校验
   - 非法迁移可识别并进入关闭路径
   - 读写超时、空闲超时接入连接生命周期

2. **资源边界控制**
   - 输入/输出缓冲区上限
   - 背压机制（读写节流）
   - 避免单连接内存无上限增长

3. **统一错误处理**
   - 项目级 `WebError`
   - `Error` 包装系统错误并保留上下文
   - 关闭路径不再分散在多个隐式分支

4. **配置与可观测性**
   - `ServerConfig` 支持配置读取、基础校验、热重载框架
   - `ServerMetrics` 支持运行指标统计
   - 结构化日志链路可输出上下文信息

5. **请求安全校验**
   - `RequestValidator` 已进入主请求链路
   - 方法白名单、路径规范化、头部和请求体大小限制
   - 非法请求可返回对应错误码

6. **HTTP/1.1 增强**
   - Keep-Alive 管理能力已接入
   - 条件请求处理模块（ETag/If-Modified-Since）已接入
   - 静态根目录默认从 `./www` 演进为 `./public`

### 正在打磨

- HTTP/2 当前是“基础模块已接入”的状态，仍需更系统的协议回归与稳定性验证。
- 性能工程（内存池、I/O 批处理、基准回归）属于 v7.5+ 重点。

## v7 请求处理主链路（当前）

```text
accept -> dispatch(SubReactor)
      -> EPOLLIN
      -> HttpRequest parse
      -> RequestValidator validate
      -> HttpResponse build
      -> send(header/body)
      -> keep-alive ? reuse : close
```

关键原则：

1. 不信任输入：先校验后处理  
2. 不隐式共享状态：一个 fd 只属于一个 EventLoop  
3. 不丢失错误上下文：所有失败路径必须可审计

## v7 常见故障与排查方向

### 1) 解析成功但响应异常

- 优先检查 `HttpRequest` 解析结果（method/version/headers/path）
- 再看 `RequestValidator` 返回码和标准化路径
- 最后确认 `HttpResponse::Init` 是否拿到正确 static root

### 2) 高并发下连接异常关闭

- 看是否触发缓冲区上限或超时路径
- 检查是否出现跨线程调用连接对象（线程亲和性）
- 对照结构化日志中连接状态迁移是否存在非法跳转

### 3) 测试超时但无崩溃

- 常见是活锁/等待条件不满足
- 优先看 `test_logs/<target>.log` 的最后有效输出
- 用 `tools.py debug --target <test_name>` 进入调试定位阻塞点

## v7 推荐验证清单

1. **功能验证**
   - 基础静态资源请求
   - Keep-Alive 多次请求复用
   - 条件请求（304 命中）
   - 非法路径/超大头部/非法方法拦截

2. **稳定性验证**
   - 长连接 + 并发请求
   - 超时连接自动清理
   - 异常请求持续注入后服务可恢复

3. **可观测性验证**
   - 指标数据随请求增长
   - 结构化日志能关联错误路径
   - 审计报告可读且可定位失败根因

## v7.5+ 下一步（明确优先级）

1. **P0：性能治理**
   - 建立压测基线（QPS、P95/P99、内存）
   - 引入性能回归门槛，避免“功能变强但性能退化”

2. **P1：协议完备性**
   - 完善 HTTP/2 关键路径测试
   - 补齐边界请求与异常帧处理验证

3. **P2：扩展性**
   - 插件化边界清晰化
   - 保持核心网络路径简洁，扩展能力外置

---

## v7 总结

v7 的价值在于：你不再只是在“写一个网络程序”，而是在构建一个能持续演进的服务器内核。

它的关键成果不是某一个 API，而是这条工程闭环：

**状态机 -> 边界控制 -> 错误体系 -> 配置观测 -> 安全校验 -> 回归验证**

这条闭环跑通，后续无论做 HTTP/2 深化、反向代理还是插件系统，复杂度都会可控。
